# AGENTS.md - jjit 多轮迭代开发规范

## 概述

本文档定义了 jjit 项目的多轮迭代开发、测试和优化流程。目标是通过至少 5 轮迭代，让输出和使用效果达到顶尖水准。

## 开发原则

- **不赶时间**：仔细彻底地处理每一个问题，确保质量而非速度
- **先读文档**：在修改涉及外部工具（如 jj、vcs-runner）的逻辑前，必须先阅读相关参考文档
- **问题驱动**：每轮迭代针对新发现的具体问题进行修复和优化
- **充分测试**：每个修复都需要经过完整测试验证
- **版本号一致性**：每次修改 `Cargo.toml` 中的 `version` 后，必须立即运行 `cargo check` 更新 `Cargo.lock`，并确保 `Cargo.lock` 的变更与版本号变更一同提交。否则 `cargo publish --dry-run` 会因未提交的 `Cargo.lock` 变更而失败。

## 开发流程（每轮）

### 1. 代码变更
- 实现功能或修复问题
- 确保代码通过 `cargo check` 和 `cargo build --release`

### 2. jj 提交
- 使用 `jj describe` 记录当前变更
- 提交信息遵循约定式提交规范

### 3. 创建示例项目
- 初始化 jj 仓库
- 创建真实代码变更用于测试

### 4. 执行测试（三种类型）

#### 单元测试
- 测试 XML 解析器（各种边界情况）
- 测试配置管理
- 测试 LLM 输出解析
- 测试 jj 命令封装

#### 自动化集成测试
- 使用真实 API Key 调用各命令
- 验证输出格式和内容
- 测试错误处理
- 测试重试机制

#### Agent 手工测试
- 模拟真实使用场景
- 测试各种边界情况
- 评估用户体验
- 检查输出质量

### 5. 制定问题报告
- 记录发现的所有问题
- 分类：致命错误 / 功能缺陷 / 体验问题 / 优化建议
- 编号：TURN1.md, TURN2.md, ...

### 6. 制定优化计划
- 针对每个问题提出解决方案
- 确定优先级
- 分配实现任务

### 7. 实施优化
- 按优先级修复问题
- 重新测试验证

### 8. 重复迭代
- 至少执行 5 轮
- 每轮都必须有新的改进

## 测试规范

### 单元测试要求

```rust
// tests/unit/
- xml_parser_tests.rs      // XML 解析各种边界情况
- config_tests.rs          // 配置读写
- jj_util_tests.rs         // jj 命令封装
- llm_output_tests.rs      // LLM 输出解析
```

### 集成测试要求

```rust
// tests/integration/
- commit_integration.rs    // commit 命令完整流程
- goto_integration.rs      // goto 命令完整流程
- pack_integration.rs      // pack 命令完整流程
- config_integration.rs    // config 命令完整流程
```

### Agent 手工测试清单

#### commit 命令
- [ ] 无变更时正确提示
- [ ] 单文件变更生成正确描述
- [ ] 多文件变更生成正确描述
- [ ] 包含 body 的提交
- [ ] 约定式提交格式正确
- [ ] 中文描述支持
- [ ] XML 格式容错
- [ ] 网络错误重试

#### goto 命令
- [ ] 根据文件历史检出

- [ ] 根据描述检出
- [ ] 找不到时正确提示
- [ ] 使用 revision 别名（@- 等）

#### pack 命令
- [ ] 组合今天提交
- [ ] 组合特定范围
- [ ] 生成正确描述
- [ ] 找不到时正确提示

#### config 命令
- [ ] set 用户级配置
- [ ] set 项目级配置

## 质量标准

### 代码质量
- 无编译警告
- 错误处理完善
- 日志输出清晰
- 代码文档完整

### 功能质量
- 所有命令正常工作
- 边界情况处理正确
- 错误提示友好
- 重试机制有效

### 用户体验
- 输出简洁明了
- 操作反馈及时
- 错误信息有用
- 配置方式直观

- 集成测试自动读取
- 绝不将 Key 提交到仓库

## 可观测性要求

### LLM 思考过程显示
- **默认行为**：LLM 的思考过程（reasoning content）应以 gray（灰色）前景色显示
- **配置项**：`show_thinking` - 是否显示思考过程（默认 `true`）
- **关闭方式**：`jjit config set show_thinking false` 或 `--no-thinking` 参数

### 提示词调试
- **参数支持**：所有命令支持 `--show-prompt` 参数
- **功能**：打印完整的 system prompt 和 user prompt，便于调试和分析 LLM 行为
- **输出格式**：
  ```
  === System Prompt ===
  [prompt content]
  
  === User Prompt ===
  [prompt content]
  
  === LLM Response ===
  [response content]
  ```

### 运行细节可观测性
- **日志级别**：支持 `--debug` 参数显示详细运行日志
- **API 调用详情**：显示请求 URL、模型、token 使用量等
- **jj 命令执行**：显示执行的 jj 命令和输出
- **XML 解析过程**：显示原始 XML 和解析结果

## 最终目标

经过至少 5 轮迭代后：
1. 代码质量达到生产级别
2. 所有命令稳定可靠
3. 用户体验流畅自然
4. 错误处理完善
5. 文档完整清晰
6. 测试覆盖全面
7. **运行过程完全可观测，便于调试和分析**
